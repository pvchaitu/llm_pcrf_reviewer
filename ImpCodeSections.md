### 1. Are we updating model weights for the model being loaded?
**YES**, the codebase is actively updating model weights during the experiment. 

The system loads a frozen **Baseline Model** and then creates a cloned **Candidate Model** where gradients are enabled. The candidate model's weights are then fine-tuned via Supervised Fine-Tuning (SFT) Regularization. 

Here is exactly where this happens in the code:

**A. Enabling Gradients for the Candidate Model**
* **File:** `run_experiment.py`
* **Method:** `main()`
* **Lines of Code:** (Approx. lines 260-264)
```python
model_candidate = AutoModelForCausalLM.from_pretrained(pcrf_config.model_cfg.model_name)
model_candidate.load_state_dict(model_base.state_dict())
model_candidate.to(pcrf_config.model_cfg.device)
for param in model_candidate.parameters():
    param.requires_grad = True # <--- Weights are explicitly unfrozen here
logger.info("FP32 candidate model loaded securely for optimization.")
```

**B. The Actual Weight Update (Training Loop)**
* **File:** `pcrf_modeling.py`
* **Class & Method:** `DerivativeRegularizer.run_standalone()`
* **Lines of Code:** 
  * *Optimizer Initialization* (Line ~622): 
    ```python

    ```
  * *Training Step / Weight Update* (Lines ~708-710):

        torch.nn.utils.clip_grad_norm_(optimized_params if optimized_params else model.parameters(), max_norm=1.0)

### 2. How the model has different generation outputs for baseline vs candidate vs served?

Because the `model_candidate`'s weights are updated (as proven above), it diverges from `model_base`. However, the **Served** output is *not* generated by a third pass of a model. Instead, it is the result of programmatic **routing and safety masking**. 

Here is exactly how the three distinct outputs are formed and transformed in the code:

#### Step 1: Baseline Output Generation
* **Where:** `run_experiment.py` inside `main()` (Lines ~158-159)
* **Code:** 
  ```python
  base_seen_metrics = EvaluatorPlus.evaluate_dataset_detailed(model_base, tokenizer, ...)
  ```
* **Explanation:** The system generates text using the completely frozen, untouched `model_base`. This populates `baseline_output`.

#### Step 2: SFT Candidate Output Generation
* **Where:** `run_experiment.py` inside `main()` (Lines ~315-316)
* **Code:**
  ```python
  reg_seen_metrics = EvaluatorPlus.evaluate_dataset_detailed(model_candidate, tokenizer, ...)
  ```
* **Explanation:** *After* the SFT training loop updates `model_candidate`'s weights in Scenario D, the system does a second generation pass. Because the weights have changed via `optimizer.step()`, `candidate_output` differs from `baseline_output`.

#### Step 3: Served Output Transformation (The Safety Router)
The final `served_output` seen in your reports (often displaying the `⚠️ Hallucination Risk Detected...` warning) is constructed by evaluating the risks of both models and choosing the safest string. 

**A. Decision Making:** 
* **File:** `pcrf_governance.py`
* **Method:** `ProtectedRouter.route_inference()` (Lines ~438-472)
* **Explanation:** The router looks at the exact match status and hallucination risk scores. It returns a decision: `"use_baseline"`, `"use_candidate"`, or `"abstain_safe_fallback"`.

**B. Output Assignment & Masking:**
* **File:** `run_experiment.py` inside `main()` (Lines ~420-430)
* **Explanation:** This is where the actual output string is chosen or overwritten.
```python
is_unresolved = (b_item['correct'] == 0 and c_item['correct'] == 0)

if is_unresolved or r_item["router_decision"] == "abstain_safe_fallback":
    # Transformation 1: Hardcoded safety overwrite
    r_item['served_output'] = SAFETY_WITHHELD_RESPONSE 
    r_item['served_status'] = "SAFETY_WITHHELD"
else:
    # Transformation 2: Select either candidate or baseline string
    r_item['served_output'] = (
        b_item["actual"]
        if r_item['router_decision'] == 'use_baseline'
        else c_item["actual"]
    )
    r_item['served_status'] = "ANSWERED"
```

**C. Redundant Artifact Masking:**
* **File:** `pcrf_dataset.py`
* **Method:** `get_public_served_output()` (Lines ~65-80)
* **Explanation:** To ensure reports are clean for customers, a secondary safety wrapper replaces any hallucinated candidate or baseline string that slipped through with `SAFETY_WITHHELD_RESPONSE`. This is why your `PCRF_Executive_Reliability_Report_200.md` shows the text `"⚠️ Hallucination Risk Detected — Response Withheld for Safety"` whenever both models fail target capture.
